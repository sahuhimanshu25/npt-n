## 🕸️ Graph Fundamentals

This tutorial teaches how to convert Twitter follower data into a graph structure and perform basic social network analysis using tools like twecoll and Gephi.

A graph is a data structure made up of **nodes** (entities) and **edges** (relationships). In social media, nodes represent users, pages, or groups, while edges represent connections such as "follows", "likes", or "friends".

|Component|Social Media Meaning|Twitter Example|
|---|---|---|
|**Node**|An entity (user, page, group)|A Twitter user account|
|**Edge**|A relationship between two entities|A "follows" connection|
|**Directed Edge**|One-way relationship (A → B ≠ B → A)|User A follows User B|
|**Undirected Edge**|Two-way relationship (A ↔ B)|Mutual friendship (not default on Twitter)|

For a Twitter followee network, each node is a user you follow (followee). A directed edge from user A to user B means **A follows B**. This structure helps visualize and analyze how users are connected, who is influential, and how information might flow.

> ⭐ **Exam Point:** In a Twitter follower graph, a **directed edge from A to B** means **A follows B** — direction matters!

```
┌─────────────────────────────────────────────────────┐
│         Twitter Followee Network Structure           │
├─────────────────────────────────────────────────────┤
│                                                     │
│   User A ───follows───▶ User B                      │
│   User A ───follows───▶ User C                      │
│   User B ───follows───▶ User C                      │
│   User C ───follows───▶ User A                      │
│                                                     │
│   Nodes: {A, B, C}                                  │
│   Directed Edges: {A→B, A→C, B→C, C→A}             │
│                                                     │
│   A's out-degree = 2  (follows 2 users)             │
│   C's in-degree  = 2  (followed by 2 users)         │
│   C→A creates a reciprocal link (A also follows C?) │
│   — Not necessarily: A→C exists, C→A exists,        │
│     but B→A does NOT exist                          │
└─────────────────────────────────────────────────────┘
```

## 📐 Ways to Represent a Graph

Computers need structured formats to understand graphs. The tutorial covers three common methods:

### Adjacency Matrix

A square 2D matrix (size = number of nodes). If there is an edge from node i to node j, the cell at row i, column j is 1; otherwise 0. It is simple to create in programming but becomes **very sparse** (mostly zeros and memory-wasting) for large social networks with millions of nodes but relatively few connections.

```
┌─────────────────────────────────────────────────┐
│         Adjacency Matrix Example                │
│         (Directed Graph: A, B, C, D)            │
├─────────────────────────────────────────────────┤
│                                                 │
│   Edges: A→B, A→C, B→C, C→A                    │
│                                                 │
│          To:  A    B    C    D                   │
│   From A  │   0    1    1    0                   │
│   From B  │   0    0    1    0                   │
│   From C  │   1    0    0    0                   │
│   From D  │   0    0    0    0                   │
│                                                 │
│   Problem: For 54 million users, this matrix    │
│   would be 54M × 54M = 2.9 × 10^15 cells       │
│   → Mostly zeros → Extremely wasteful           │
└─────────────────────────────────────────────────┘
```

> ⭐ **Exam Point:** Adjacency Matrix is **inefficient for large social networks** because it becomes **very sparse and wastes memory** (mostly zeros).

### GraphML

An XML-based format. It uses `<graph>`, `<node>`, and `<edge>` tags. Each node has a unique ID, and each edge specifies source and target nodes. This is the format generated by twecoll and imported into Gephi.

```
┌─────────────────────────────────────────────────┐
│         GraphML Structure (Simplified)          │
├─────────────────────────────────────────────────┤
│                                                 │
│   <graphml>                                     │
│     <graph edgedefault="directed">              │
│       <node id="A"/>                            │
│       <node id="B"/>                            │
│       <node id="C"/>                            │
│       <edge source="A" target="B"/>             │
│       <edge source="A" target="C"/>             │
│       <edge source="B" target="C"/>             │
│     </graph>                                    │
│   </graphml>                                    │
│                                                 │
│   → XML-based, human-readable, attribute-rich   │
│   → Used by twecoll output and Gephi input      │
└─────────────────────────────────────────────────┘
```

### Edge List / CSV

A simple list showing source → target relationships. Easy to generate and read but less rich for complex attributes.

```
┌─────────────────────────────────────────────────┐
│         Edge List / CSV Example                 │
├─────────────────────────────────────────────────┤
│                                                 │
│   source, target                                │
│   A, B                                          │
│   A, C                                          │
│   B, C                                          │
│   C, A                                          │
│                                                 │
│   → Simple, lightweight, easy to generate       │
│   → Less rich for storing node/edge attributes  │
└─────────────────────────────────────────────────┘
```

## ⚖️ Comparison: Graph Representation Methods

|Dimension|Adjacency Matrix|GraphML|Edge List / CSV|
|---|---|---|---|
|**Format**|2D square matrix (0s and 1s)|XML-based with tags|Simple text list (source, target)|
|**Readability**|Difficult for large graphs|Human-readable (XML)|Very easy to read|
|**Memory Efficiency**|Poor for sparse graphs (mostly zeros)|Good (stores only existing edges)|Good (stores only existing edges)|
|**Attribute Support**|Limited (only edge existence)|Rich (supports node/edge attributes)|Limited (basic source-target only)|
|**Tool Compatibility**|General programming (numpy, etc.)|Gephi, twecoll, NetworkX|Most tools, spreadsheets|
|**Scalability**|Very poor for millions of nodes|Good|Good|
|**Use Case**|Small, dense graphs; academic examples|Standard interchange format; network analysis tools|Quick prototyping; data export|
|**Generated by twecoll**|No|Yes (`.gml` file)|No (but can be derived)|

> ⭐ **Key Takeaway:** For large social networks like Twitter (54M+ users), **GraphML and Edge List** are far more practical than Adjacency Matrix due to sparsity.

## 🔧 Collecting Twitter Follower Network Data (twecoll)

The tool **twecoll** (a command-line Python script) is used to fetch Twitter network data and convert it into a graph format for analysis.

```
┌──────────────────────────────────────────────────────────────┐
│              twecoll Data Collection Pipeline                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 1: INIT — Authorization                                │
│          │  Authorize using your Twitter app's               │
│          │  Consumer Key and Secret                          │
│          │  Enter the PIN provided by Twitter                │
│          │                                                   │
│          ▼                                                   │
│  Step 2: FETCH — Download Network Data                       │
│          │  Download your followees (friends)                │
│          │  Download their followees (friends-of-friends)    │
│          │                                                   │
│          ▼                                                   │
│  Step 3: EDGELIST — Generate Graph                           │
│          │  Command: twecoll edgelist <username>             │
│          │  Generate the connections                         │
│          │  Output: .gml file (GraphML format)              │
│          │                                                   │
│          ▼                                                   │
│  Result: Directed graph of your personal Twitter network     │
│          ready for import into Gephi                         │
└──────────────────────────────────────────────────────────────┘
```

|twecoll Command|Purpose|Output|
|---|---|---|
|`twecoll init`|Authorize with Twitter API using Consumer Key/Secret + PIN|Authentication tokens|
|`twecoll fetch <username>`|Download followees and friends-of-friends|Raw follower/following data|
|`twecoll edgelist <username>`|Generate edge list and graph file|`.gml` file (GraphML format)|

> ⭐ **Exam Point:** The `twecoll edgelist <username>` command generates the **edge list and .gml file**. The final output format is **`.gml` (GraphML)**.

## 📊 Basic Social Network Analysis (SNA) Metrics

These metrics help understand the structure and importance of nodes within a social network graph.

### Degree

|Metric|Definition|Twitter Meaning|
|---|---|---|
|**In-degree**|Number of edges coming INTO a node|Number of followers|
|**Out-degree**|Number of edges going OUT from a node|Number of accounts you follow|
|**Total Degree**|In-degree + Out-degree|Total connections (both directions)|

```
┌─────────────────────────────────────────────────┐
│              Degree Calculation Example          │
├─────────────────────────────────────────────────┤
│                                                 │
│   Node X:                                       │
│     Incoming edges (followers): 5               │
│     Outgoing edges (following): 3               │
│                                                 │
│     In-degree  = 5                              │
│     Out-degree = 3                              │
│     Total Degree = 5 + 3 = 8                    │
└─────────────────────────────────────────────────┘
```

### Centrality (Measures of Importance)

|Centrality Type|What It Measures|Identifies|Twitter Interpretation|
|---|---|---|---|
|**In-degree Centrality**|Number of incoming connections|Most followed / influential nodes|Users with the most followers|
|**Betweenness Centrality**|How often a node lies on shortest paths between others|Important bridges or connectors between groups|Users who connect otherwise separate communities|
|**Closeness Centrality**|Average shortest path distance to all other nodes|Nodes that can reach others most quickly|Users with the most efficient access to the whole network|

```
┌──────────────────────────────────────────────────────────────┐
│            Centrality Types — Visual Intuition                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  IN-DEGREE CENTRALITY (Most Popular)                         │
│                                                              │
│       B ──▶ ┌───┐ ◀── D                                     │
│       C ──▶ │ A │ ◀── E       A has highest in-degree        │
│       F ──▶ └───┘ ◀── G       → Most followers / influential │
│                                                              │
│  BETWEENNESS CENTRALITY (Bridge / Connector)                 │
│                                                              │
│   [Cluster 1] ──▶ X ──▶ [Cluster 2]                         │
│                                                              │
│       X sits on the only path between two groups             │
│       → Key bridge / information bottleneck                  │
│                                                              │
│  CLOSENESS CENTRALITY (Most Central / Reachable)             │
│                                                              │
│           B ── A ── C                                        │
│                │                                             │
│                D                                             │
│                                                              │
│       A has shortest average distance to B, C, D             │
│       → Can reach everyone most quickly                      │
└──────────────────────────────────────────────────────────────┘
```

> ⭐ **Exam Point:** **Betweenness Centrality** identifies nodes that act as **bridges between different groups** — they lie on many shortest paths between other nodes.

### Modularity

Modularity measures how strongly the graph is divided into **communities** (groups of densely connected nodes). Higher modularity means clearer community structure.

|Modularity Value|Interpretation|
|---|---|
|**High (close to 1)**|Strong community structure — clear, well-separated groups|
|**Low (close to 0)**|Weak community structure — connections are evenly distributed|
|**Negative**|Nodes are less connected within groups than expected by chance|

> ⭐ **Exam Point:** **High modularity** indicates **strong community structure** — the network has clearly defined clusters of densely connected nodes.

## 🖥️ Visualization with Gephi

Gephi is a popular open-source tool for exploring and visualizing graphs. It turns raw Twitter data into an understandable visual map showing influential users, clusters, and overall network structure.

|Gephi Feature|Purpose|
|---|---|
|**File Import**|Open `.gml` file generated by twecoll|
|**Layout Algorithms**|Arrange nodes visually (e.g., Fruchterman-Reingold, ForceAtlas)|
|**Statistics Panel**|Compute degree distribution, modularity, centrality, etc.|
|**Appearance Panel**|Color nodes by modularity class; size nodes by degree|
|**Labels**|Show node labels (usernames) for identification|
|**Preview Tab**|Prepare final visualization for export|
|**Export**|Save as PNG, SVG, or PDF|
|**Data Laboratory Tab**|View and edit raw node and edge data in table format|

> ⭐ **Exam Point:** The **Fruchterman-Reingold** algorithm in Gephi is a **force-directed layout** — nodes repel each other, but connected nodes attract, making the graph readable.

## 📖 Key Terminology

|Term|Definition|
|---|---|
|**Node**|An entity in a graph (user, page, group)|
|**Edge**|A relationship/connection between two nodes|
|**Directed Edge**|A one-way edge (A→B does not imply B→A)|
|**Self-loop**|An edge from a node to itself|
|**In-degree**|Number of incoming edges to a node (followers on Twitter)|
|**Out-degree**|Number of outgoing edges from a node (following on Twitter)|
|**Total Degree**|In-degree + Out-degree|
|**Adjacency Matrix**|2D matrix representation of a graph; 1 = edge exists, 0 = no edge|
|**GraphML**|XML-based standard format for representing graphs with attributes|
|**Edge List / CSV**|Simple text list of source→target pairs|
|**twecoll**|Command-line Python tool for collecting Twitter network data|
|**Gephi**|Open-source graph visualization and analysis software|
|**In-degree Centrality**|Measure of importance based on number of incoming connections|
|**Betweenness Centrality**|Measure of importance based on lying on shortest paths between other nodes|
|**Closeness Centrality**|Measure of importance based on average shortest distance to all other nodes|
|**Modularity**|Measure of how strongly a graph is divided into communities|
|**Fruchterman-Reingold**|Force-directed layout algorithm used in Gephi (nodes repel, connected nodes attract)|
|**Data Laboratory**|Gephi tab for viewing/editing raw node and edge data in table format|
|**.gml file**|Output format of twecoll; GraphML file imported into Gephi|
